Case Study: Zero‑Budget Inventory Intelligence – Identifying ₦1.2M–₦2M Monthly Expiry Risk

The Business Problem: Penafort was writing off goods worth ₦1.2–2 million every month due to product expiry. Their existing ERP could not accurately track expiring items across 36,000+ SKUs, nor monitor stock levels (surplus or low stock) to advise on transfers or procurement. The result: preventable losses and missed opportunities to rebalance inventory.

The Zero‑Budget Solution: I engineered a production system using only free tiers: Streamlit for dashboard, Supabase (PostgreSQL) for database and scheduled jobs. No ad spend, no paid tools.

🔍 Expiry Tracking (90/120/180 day windows)

  • Scans all 36k+ SKUs daily, flagging products that expire in 90, 120, or 180 days (colour‑coded).
  • Calculates the total value of expiring stock – typically between ₦1.2M and ₦2M monthly.
  • Daily email at 10am Nigeria time lists nearest‑expiry items first, with quantities, batch numbers, and monetary value at risk.
  • Advises inter‑branch transfer for expiring stock that could sell faster elsewhere, or discount/return if unsellable.

📊 Stock Limit Monitoring (Surplus & Low Stock)

  • Configurable min/max thresholds per SKU (e.g., min = 100 units, max = 500 units).
  • Low stock alert: When current stock ≤ min threshold, the system advises procurement to reorder (suggested quantity = max − current).
  • Surplus alert: When current stock ≥ max threshold, the system flags surplus and recommends transferring excess to other branches (with lower stock) or running a promotion.
  • Cross‑branch rebalancing suggestions: If Branch A has surplus and Branch B has low stock of the same SKU, the email recommends a specific transfer quantity.
  • All alerts appear in the same daily 10am email – stakeholders see expiry, surplus, and low stock in one place.

⚙️ Production Hardening on Free Infrastructure

  • All expiry and stock limit logic runs inside PostgreSQL (views, functions, pg_cron) – Streamlit is only for on‑demand dashboard views.
  • pg_cron triggers daily at 10am WAT: runs expiry scan, compares stock vs min/max thresholds, generates transfer/procurement advice, and sends emails via pg_net / edge functions.
  • Chunked CSV uploads and paginated tables handle 36k+ rows without memory crashes on Streamlit Cloud’s free tier.
  • Atomic RPC functions ensure data integrity across concurrent branch uploads.

📈 Outcome

  • The system accurately identifies expiring products worth ₦1.2M–₦2M monthly, giving stakeholders clear visibility into at‑risk value.
  • Procurement receives daily low stock alerts – they decide when and how much to reorder.
  • Branch managers get surplus and expiry‑based transfer suggestions – they execute inter‑branch moves as needed.
  • All on $0 monthly infrastructure cost – a true example of budget‑cutting production engineering.
Skills & Tools used: Python (Streamlit), Supabase (PostgreSQL, pg_cron, Edge Functions/pg_net), Pandas, Email automation.

Case Study: Real‑Time Market Penetration Intelligence – ₦1.7M Campus Campaign

The Problem: A brand was spending ₦1.7 million to penetrate the LASU student market (SOM = 2,500 students). They had no live visibility into which faculties were converting, no real‑time CPA tracking, no way to model market potential, no memory of past strategic decisions, and siloed advice.

The Zero‑Budget Solution: I built a production system using free tiers: Streamlit (dashboard), Supabase (database + pgvector + RAG), Gemini (embeddings), and Groq (Llama) (reasoning). The system is generic – it works for any niche (fragrance, supermarket, etc.).

  • Registers students live by faculty – sidebar form adds name + faculty (Arts, Science, Law, etc.) to leads table instantly.
  • Uses TAM‑SAM‑SOM stored in the database – SOM = 2,500 students. Target CPA = ₦1.7M ÷ 2,500 = ₦680. Actual CPA is calculated live from budget ÷ leads.
  • Pipeline value = total leads × ₦15,000 (estimated student expenditure) – displayed automatically.
  • Tracks loyalty points per student – loyalty_points column records spending, enabling retention strategies.
  • Intent classifier – Llama routes queries to: Customer analysis (JTBD, CJM), Market analysis (VRIO, Blue Ocean), or Hybrid optimisation (Ansoff, resource allocation).
  • RAG (Retrieval‑Augmented Generation) – Gemini embeds questions; Supabase vector search (pgvector) retrieves similar past memories; Llama answers using those memories + live metrics. The system learns from itself.
  • Automated email alerts when CPA exceeds ₦680 – If actual CPA > ₦680, an email is sent with Nigeria‑specific, actionable recommendations (e.g., shift budget to WhatsApp marketing, run LASU referral campaigns).
  • Human‑approved strategy proposals – AI suggestions go into suggestions table as pending; stakeholders approve or reject – no blind automation.

⚙️ Production on Free Infrastructure

  • All data (leads, loyalty points, memories, TAM‑SAM‑SOM) in Supabase free tier.
  • Vector search (pgvector) and RAG inside PostgreSQL.
  • Streamlit Cloud hosts dashboard – $0 cost.
  • Groq (Llama) and Gemini APIs on free tiers.
  • Email alerts via SMTP (free, Gmail).

📈 Measurable Impact

  • Live CPA tracking against the ₦680 target – email alerts prevent wasteful spending.
  • Pipeline value automatically calculated (leads × ₦15,000) – no spreadsheets.
  • Department‑level registration allows reallocation of ₦1.7M budget to high‑converting faculties mid‑campaign.
  • RAG memory means past decisions are retrieved and reused – no repeated analysis.
  • All AI proposals require human approval – safe, accountable automation.
Skills & Tools used: Streamlit, Supabase (pgvector, RPC, triggers), Gemini (embeddings), Groq (Llama 3.1/3.3), Python, TAM‑SAM‑SOM, loyalty tracking, intent classification, email automation.

Case Study: MZIMS – Custom ERP for E‑commerce (Solving Sales Tracking & Operational Blindness)

The Problem: A fragrance e‑commerce brand (MuzoScents) had been operating but sales tracking was nearly impossible. Spreadsheets and manual records meant they knew only a minimum of ₦900k in revenue had been generated – the actual figure was certainly higher, possibly significantly, because transactions were lost, untracked, or misrecorded. They had no real‑time inventory visibility, no profit tracking, no way to identify expiring stock, and monthly reporting was a painful manual guess.

The Zero‑Budget Solution: I built a production ERP system using free tiers: Streamlit (interface), Supabase (database), and SMTP (email reports). The system runs on $0 infrastructure and solves the tracking problem by providing:

  • Sales ledger with atomic transactions – A PostgreSQL function checks stock (FEFO), deducts inventory, calculates profit, and logs the sale in one operation – no lost or partial transactions.
  • Complete sales history – Every sale is timestamped, linked to product/batch, with quantity, revenue, COGS, and profit. The business now knows exactly what has been sold, when, and at what margin.
  • Intelligent purchasing advice – Daily recomputed from sales velocity (avg daily demand). Advises when to reorder (stock below reorder point or <14 days of stock left), with suggested quantities.
  • Expiry risk & FEFO (First Expired First Out) – Scores inventory batches by days to expiry and financial exposure. Suggests consumption order to minimise waste.
  • Automated monthly email report – Emails actual sales summary (revenue, COGS, profit, margin, top products) – finally giving stakeholders reliable, auditable numbers.
  • Built‑in analytics – Daily sales/profit trends, top products, slow‑moving stock (>60 days of inventory), category performance.
  • Bulk CSV upload – For products, inventory, and sales – chunked to avoid memory limits.

⚙️ Production on Free Infrastructure

  • Streamlit Cloud – $0 frontend
  • Supabase (PostgreSQL + auth + pg_cron) – free tier
  • SMTP (Gmail) – free email reports
  • Daily demand recompute via pg_cron (scheduled in DB)

📈 Measurable Impact

  • Solved the tracking problem – Every sale is recorded atomically. The business can finally see its true revenue and profit in real time.
  • The previous minimum estimate of ₦900k is no longer a guess; actual numbers are now available on demand.
  • Automated purchasing advice reduced stockouts and excess inventory.
  • Monthly reports eliminated manual spreadsheet work.
  • FEFO recommendations reduced expiry waste.
  • All transaction integrity ensured via atomic database functions.
Skills & Tools used: Streamlit, Supabase (PostgreSQL, RPC, triggers, pg_cron), Pandas, SMTP, Python, FEFO logic, demand‑based purchasing.

Case Study: Kevs Digital Business Concierge – Cross‑Device Conversational AI for Retail & E‑commerce

The Problem: Retailers (physical stores + e‑commerce) receive customer queries across WhatsApp and their website – but support is fragmented. Customers expect to ask about inventory, loyalty, order status, and business info (location, opening times, offers). Without a unified system, support is manual, context is lost across devices, and customers get frustrated.

The Solution – Modular, RAG‑powered Concierge with Cross‑Device Identity: I built a production‑grade conversational AI system using a modular architecture (FastAPI backend, Supabase pgvector for memory, Gemini embeddings, Groq Llama 3.3, WhatsApp Cloud API). All on free tiers. The system is live, working, and available for any organisation to adopt (productised commodity).

  • Unified retail support – Real‑time access to inventory, loyalty points, order history, and business knowledge (location, opening hours, offers, policies).
  • General business knowledge – Static business info stored in Supabase. AI answers “What time do you close?” or “Do you have a discount code?” from this knowledge base.
  • Cross‑device user tracking – WhatsApp uses phone number; web chat uses email authentication. Both link to the same customer profile – AI remembers conversations and preferences across channels.
  • WhatsApp Gateway – Webhooks receive messages; AI responds with access to all modules.
  • Web Chat Widget – Embedded HTML/CSS/JS widget (in Hugging Face demo) authenticates via email, then queries the same FastAPI backend.
  • RAG (Retrieval‑Augmented Generation) – Gemini embeds queries; Supabase pgvector retrieves relevant memories (past conversations, inventory, loyalty, business info). Llama 3.3 answers grounded in those memories + live data.
  • Loyalty Module – Tracks points, redemption, referrals. AI answers “How many points do I have?” across devices.
  • Inventory Module – Real‑time lookups to inventory tables. AI answers “Do you have X in stock?” and suggests alternatives.
  • Human hand‑off (escalation) – When the AI cannot answer a question (low confidence, out‑of‑scope, or sensitive issue), the conversation is seamlessly escalated to a human customer service agent via WhatsApp or an internal dashboard.
  • Demo Mode – Hugging Face Space includes a chat widget simulating both channels – perfect for portfolio review and client testing.

⚙️ Production on Zero Budget

  • Hugging Face Spaces – free tier hosts FastAPI backend (Docker).
  • Supabase – free tier (PostgreSQL, pgvector, auth, real‑time).
  • Groq (Llama 3.3) – free API tier.
  • Gemini – free tier for embeddings.
  • WhatsApp Cloud API – free for testing (demo mode avoids live calls).

🧩 Why it's complex (modular design + cross‑device identity + multi‑domain knowledge + human hand‑off)

  • Each module (WhatsApp, web widget, RAG, loyalty, inventory, business info, escalation) runs independently but shares the same vector DB, LLM orchestrator, and customer identity table (phone ↔ email mapping).
  • Business info is stored as structured data (hours, location, offers) and also embedded for semantic retrieval.
  • Human hand‑off logic is integrated: confidence scoring triggers real‑time notifications to agents, and the conversation context is preserved.
  • Designed to be extended (add email channel, POS) without rewriting core logic.
  • All secrets via environment variables – safe for open‑source portfolio (MIT license).

📌 Current Status

  • Live demo on Hugging Face Spaces.
  • Code available on GitHub (MIT license).
  • Ready for adoption by any retail or e‑commerce organisation – no metrics claimed as it's a productised commodity.
Skills & Tools used: FastAPI, Supabase (pgvector, RPC, auth), Gemini (embeddings), Groq (Llama 3.3), WhatsApp Cloud API, Docker, Hugging Face Spaces, Python, RAG architecture, cross‑device authentication, business knowledge management, human‑in‑the‑loop escalation.
